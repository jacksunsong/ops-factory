## 1. 设计初衷：为什么要做这件事

### 1.1 业务背景

企业 ITSM（IT Service Management）运营每天都在产生大量工单数据——事件（Incident）、变更（Change）、服务请求（Request）、问题（Problem）。这些数据分布在不同的系统中，格式各异，是运维团队最核心的"运营资产"。

然而，数据的存在并不等于数据的价值被释放。在实际的 ITSM 运营场景中，**服务交付经理（SDM）** 面临三个核心痛点：

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **数据看不懂** | 原始工单数据缺乏聚合与解读，SDM 需要手动从多个 Excel 中拉数据、做透视表、算指标 | 获取一个 SLA 达成率就需要 30 分钟以上 |
| **维度看不到** | 固定仪表盘只能展示预设维度，无法按需钻取、交叉分析、自然语言追问 | 发现 SLA 偏低后，无法快速定位是哪类问题、哪个人员导致的 |
| **报告来不及** | 月度/季度运营报告依赖人工整理，从数据提取到报告产出周期长达数天 | 管理层拿到的报告已经滞后，错过最佳干预窗口 |

### 1.2 为什么用 AI Native 方式解决

传统的 BI 解决方案遵循"人找数据"的交互模式——用户需要理解数据结构、找到正确报表、手动筛选和解读。这种模式对 SDM 有较高的专业门槛。

我们选择 **AI Native** 范式，实现从"人找数据"到"数据找人"的转变：

```
传统 BI:  用户 → 理解数据结构 → 找到正确报表 → 手动筛选 → 人工解读 → 撰写报告
AI Native: 用户 → 自然语言提问 → Agent 理解意图 → 自动获取数据 → 生成洞察报告
```

核心转变体现在五个维度：

| 维度 | 传统 BI | AI Native BI（我们的方案） |
|------|---------|------------------------|
| 交互方式 | 点击仪表盘、拖拽筛选 | 自然语言对话 |
| 分析路径 | 预设报表、固定维度 | Agent 自主决策工具与参数 |
| 报告生成 | 人工撰写、模板填充 | Agent 基于数据自动生成 |
| 洞察深度 | 依赖分析师经验 | 内置风险雷达、异常检测、交叉关联 |
| 交付时效 | 天级 | 分钟级 |

### 1.3 整体规划目标

构建一个 **数据可计算、意图可理解、报告可交付** 的 AI Native ITSM 运营分析体系：

- **数据层**：将四类 ITSM 工单原始数据结构化、可计算化，建立统一的指标计算引擎
- **工具层**：将分析能力封装为 AI Agent 可准确调用的意图驱动工具集
- **交互层**：通过自然语言对话直接获取分析报告，零学习成本

### 1.4 为什么拆成两个模块

BI 智能分析引擎和 Report Agent 不是一个大功能，而是两个职责清晰的独立模块。这样拆分的原因是：

**两类消费者的数据需求完全不同**：

| 消费者 | 数据需求 | 交互方式 |
|--------|---------|---------|
| 人类仪表盘 | 一次性加载全量数据渲染 8 个 Tab，数据格式面向图表编码 | HTTP API → 前端渲染 |
| AI Agent | 按意图精准获取特定域、特定维度的指标，数据格式面向理解和推理 | 自然语言 → MCP 工具调用 |

如果只做仪表盘 API，Agent 每次都要加载全量数据再自行筛选——对 27B 参数模型来说，上下文窗口和推理能力会被浪费。独立的分析引擎让 Agent 像调 API 一样按需取数。

同时，两个模块的演进节奏也不同：BI 引擎需要稳定的数据契约和计算逻辑，而 Agent 需要灵活的 prompt 优化和工具迭代。解耦后可以独立演进、独立部署。

## 3. 模块一：BI 智能分析引擎

### 3.1 模块概述

BI 智能分析引擎是整个系统的 **数据底座**。它将分散在多个 Excel 文件中的 ITSM 工单原始数据，转化为结构化、可计算、可查询的指标体系，同时服务于两个消费者：**人类仪表盘** 和 **AI Report Agent**。

**代码位置**：`business-intelligence/`

**技术栈**：Java 21 + Spring Boot 3.3 + Apache POI 5.4 + Maven

### 3.2 核心能力

#### 能力 1：八域聚合指标体系

覆盖 ITSM 运营的 8 个核心分析域，每个域输出一组预计算的 KPI：

| 分析域 | API 路径 | 核心指标 | 回答的业务问题 |
|--------|----------|---------|-------------|
| **Executive** | `/metrics/executive` | 综合健康评分 (0-100)、等级、风险项 | "运营整体怎么样？" |
| **SLA** | `/metrics/sla` | 响应/解决达成率、违约数、P1/P2 达成率 | "服务承诺兑现能力如何？" |
| **Incident** | `/metrics/incidents` | 工单总量、P1/P2 占比、MTTR | "故障影响面和恢复速度如何？" |
| **Change** | `/metrics/changes` | 成功率、紧急变更占比、变更致事件率 | "变更管控成熟度如何？" |
| **Request** | `/metrics/requests` | 完成率、CSAT、平均履行时长 | "服务请求交付质量如何？" |
| **Problem** | `/metrics/problems` | 关闭率、RCA 率、已知错误数 | "根因治理能力如何？" |
| **Cross-Process** | `/metrics/cross-process` | 变更致事件率、脆弱度评分、48h P1/P2 窗口 | "跨流程协同风险如何？" |
| **Workforce** | `/metrics/workforce` | 人均吞吐、积压、SLA 达成、满意度 | "团队能力和负载均衡如何？" |

**健康评分权重模型**（贯穿 Java 和 Python 两套子系统，保持一致）：

```
综合健康评分 = Incident SLA(25%) + MTTR(15%) + P1/P2 Rate(10%)
            + Change Success(15%) + Change-Induced Incident(5%)
            + Request SLA(10%) + CSAT(10%)
            + Problem Closure(5%) + Backlog(5%)
```

**达成效果**：SDM 不再需要逐个 Excel 表拉数据算指标，一次 API 调用获得任意域的全部 KPI。

#### 能力 2：多维度下钻分析

每个域支持多个分析维度的按需展开：

| 维度类型 | 示例 | 回答的问题 |
|---------|------|----------|
| 按优先级 | P1/P2/P3/P4 各自的 SLA 达成率 | "高优先级的 SLA 为什么这么低？" |
| 按类别 | Database/Network/Application 各自的 MTTR | "哪类故障修复最慢？" |
| 按人员 | 每个 Resolver 的 SLA 达成率 | "谁负责的工单违约最多？" |
| 按时间 | 周度/月度的趋势变化 | "SLA 是在改善还是恶化？" |
| 按风险等级 | High/Medium/Low 变更的成功率 | "高风险变更的失败率是多少？" |
| 按部门 | 各部门的请求满意度 | "哪个部门对服务最不满意？" |
| 按根因 | Technical Defect / Process Gap / Human Error | "问题的根因集中在哪？" |

**达成效果**：SDM 不满足于整体数字时，可以自然语言追问下钻——"按人员看呢？""按类别拆一下？"。

#### 能力 3：时间序列趋势

支持按日/周/月粒度计算任意指标的时序数据，并支持多系列合并输出：

- SLA 趋势：同时返回响应达成率 + 解决达成率 + P1/P2 达成率三条线
- Incident 趋势：同时返回工单量 + SLA 达成率
- MTTR 趋势：同时返回整体均值 + P1/P2 均值

**达成效果**：一眼看出趋势方向，而非孤立的单点数字。支持"SLA 是在改善还是恶化？"这类时间维度的问题。

#### 能力 4：跨流程关联追溯

ITSM 四个流程不是孤立的，关联追溯能力打通了流程间的壁垒：

| 关联类型 | 说明 | 业务价值 |
|---------|------|---------|
| 变更 → 事件 | 48 小时窗口内的 P1/P2 关联 | 识别"高危变更"，量化变更风险 |
| 事件 → 问题 | 重复发生的事件归类为已知错误 | 从被动救火到主动根因治理 |
| 请求 → 事件 | 请求激增预警 | 识别服务异常的早期信号 |
| 单据追溯 | 输入工单号，自动关联上下游 | 从"看单个工单"到"看关联链路" |

**达成效果**：从"看单个工单"升级为"看关联链路"，识别系统性风险。

#### 能力 5：通用数据查询

对四类工单数据支持任意字段的过滤、排序、分页和聚合计算：

- **过滤**：等值、不等、包含、范围、IN 等操作符
- **聚合**：计数（count）、均值（avg）、求和（sum）、百分比（percentage）、分布（distribution）
- **排序**：按任意字段升序/降序
- **分页**：支持 offset + limit

**达成效果**：当预计算指标无法满足个性化分析需求时，Agent 可以用通用查询自由组合，确保任何分析问题都有解。

### 3.3 数据契约

BI 引擎定义了标准化的数据导入规范（`itil-ticket-export-schema-v1`），支持适配多种 ITSM 平台（ServiceNow、BMC Remedy、HPSM、SMAX、iMOC eTicket 等）：

| 数据文件 | 工单类型 | 核心字段 |
|---------|---------|---------|
| `Incidents-exported.xlsx` | 事件工单 | ticket_id, priority, category, status, opened_at, resolved_at, response_time_minutes, resolution_time_minutes |
| `Changes-exported.xlsx` | 变更工单 | + change_type, risk, incident_ids |
| `Requests-exported.xlsx` | 服务请求 | + satisfaction_score, requester_dept |
| `Problems-exported.xlsx` | 问题工单 | + root_cause, cause_code, workaround |
| `SLAs-exported.xlsx` | SLA 标准 | 各优先级的响应/解决时限 |

SLA 达成判定逻辑（引擎自动计算，不依赖原始数据中的标记字段）：

```
SLA Compliant = (response_time_minutes ≤ response_sla_min)
            AND (resolution_time_minutes ≤ resolution_sla_min)
```

主要能力：

### 3.6 仪表盘 8 大 Tab 设计

#### 为什么设计 8 个 Tab

8 个 Tab 的划分基于 ITIL 四大核心流程（事件、变更、请求、问题）再加两个横切维度（SLA、跨流程关联）和两个管理层视角（执行摘要、团队效能），形成 **4 + 2 + 2** 的结构：

```
管理层视角（自上而下）
├── Tab 1 执行摘要        ← SDM 管理层一眼看全貌
│
ITIL 流程域（4 大核心流程）
├── Tab 2 SLA 分析        ← 贯穿所有流程的服务承诺
├── Tab 3 事件分析        ← Incident Management
├── Tab 4 变更分析        ← Change Management
├── Tab 5 请求分析        ← Request Fulfillment
├── Tab 6 问题分析        ← Problem Management
│
横切维度（打破流程壁垒）
├── Tab 7 跨流程关联      ← 流程间的因果链路和风险传导
│
执行层视角（自下而上）
├── Tab 8 团队效能        ← 人员维度的产量、效率、质量
```

**设计依据**：

- **ITIL 4 实践框架**：事件管理、变更支持、服务请求、问题管理是 ITSM 的四大核心实践，各自有独立的分析语境和 KPI 体系，拆 Tab 符合运营团队的组织分工
- **SLA 独立成 Tab**：SLA 达成率是 SDM 与客户之间最核心的合同性指标，横跨事件和请求两个流程，需要独立的全局视角而非分散在流程 Tab 中
- **跨流程关联 Tab**：ITIL 强调流程间的协同效应。变更导致事件、事件暴露问题、问题关联请求——这些跨流程因果关系无法在单一流程 Tab 中呈现，需要独立视角
- **执行摘要 Tab**：管理层不需要钻取细节，需要一个综合健康评分 + 风险摘要 + 趋势的"仪表盘入口"
- **团队效能 Tab**：运维服务的质量最终落到人。将四类流程的指标映射到人员维度，支撑绩效管理和负载均衡决策

#### 健康评分权重模型

综合健康评分（0-100）贯穿 Tab 1 执行摘要，是整个仪表盘的核心数字。权重分配如下：

| 流程域 | KPI | 权重 | 设计依据 |
|--------|-----|------|---------|
| 事件管理 | SLA 达成率 | 25% | SLA 是服务提供商与客户之间的合同性承诺，ITIL 4 和 ISO 20000 均将其列为首要监控指标。设为最高权重体现"承诺兑现优先" |
| 事件管理 | MTTR（平均修复时间） | 15% | 直接衡量客户痛苦持续时间和运维效率。ITIL 4 事件管理实践将解决时间列为关键效率 KPI |
| 事件管理 | P1/P2 趋势 | 10% | 衡量高优工单的增减趋势，是未来 SLA 违约的先行指标。稳定 SLA 下的 P1/P2 上升意味着潜在风险积累 |
| 变更管理 | 变更成功率 | 15% | ITIL 中未授权或失败变更是事件的首要诱因。行业基准显示成熟变更流程成功率达 95%+ |
| 变更管理 | 变更致事件率 | 5% | 与变更成功率部分重叠但捕获不同信号：变更可能"成功完成"但仍引发后续事件。权重较低因为已与成功率互补 |
| 请求管理 | 请求 SLA 达成率 | 10% | 服务请求是客户日常感知的主要触点，履约时效直接影响客户满意度。ISO 20000 要求定义履行时间 |
| 请求管理 | CSAT 满意度 | 10% | 与请求 SLA 等权，平衡客观履约指标与主观感知。使用 5 分制，与行业标准 CSAT 测量方式对齐 |
| 问题管理 | 问题关闭率 | 5% | 问题管理是长周期、低频率的根因治理活动，权重低于事件/变更反映其运营紧迫性较低 |
| 问题管理 | 积压率 | 5% | 与关闭率互补，防止只关注关闭速度而忽视积压增长 |

> **事件管理占 50% 总权重**——这反映 ITSM 行业的普遍共识：事件管理是最高频、最面向客户的流程，其健康度对整体运营质量影响最大。

#### Tab 1：执行摘要

**定位**：管理层一眼看全貌的"运营健康仪表盘入口"。

**设计思路**：管理层（SDM、项目经理）不需要钻取细节，需要的是——整体好不好（健康评分）、哪里有问题（风险摘要）、趋势如何（月度趋势）。执行摘要将四个流程域的健康度压缩为单一评分和三色等级，配合 Top 5 风险项，让管理者在 30 秒内判断是否需要进一步关注。

| 内容类型 | 内容 | 业务价值 |
|---------|------|---------|
| **健康评分 Hero** | 0-100 综合评分 + Stable/Watch/Risk 等级 + 趋势变化提示 | 一句话回答"运营整体怎么样"，无需看 4 个 Tab |
| **4 流程健康度** | 事件/变更/请求/问题各域的独立评分 + 核心摘要 | 快速定位哪个流程拖后腿 |
| **风险摘要** | Critical/Warning/Attention 计数 + Top 5 具体风险项 | 从"分数"到"具体什么事"的桥梁 |
| **6 张 MetricCard** | SLA 达成率、MTTR、变更成功率、变更致事件率、请求满意度、问题关闭率 | 6 个最关键运营 KPI 的一目了然展示 |
| **月度健康趋势** | 健康评分 + P1/P2 数量的月度走势 | 判断运营是在改善还是恶化 |
| **关键关注项表格** | SLA 违约数、失败变更数、未完成请求数、未关闭问题数 | 需要立即关注的 4 项硬数字 |

#### Tab 2：SLA 分析

**定位**：深度拆解 SLA 达成情况，从整体到违约明细的完整分析链路。

**设计思路**：SLA 是 SDM 与客户之间的核心契约。单纯的"达成率 85%"不够——需要知道是响应违约还是解决违约、哪个优先级拖后腿、哪个类别/人员违约最多、趋势是在改善还是恶化。这个 Tab 按优先级 → 类别 → 人员 → 时间的维度逐层展开，最终落到违约样本明细，形成"发现问题 → 定位原因 → 追溯到单"的闭环。

| 内容类型 | 内容 | 业务价值 |
|---------|------|---------|
| **SLA Hero** | 综合/响应/解决达成率 + 违约数 + P1/P2 达成率 | 区分响应压力和解决压力，判断瓶颈在哪端 |
| **2 张 SLA 维度卡** | 响应 SLA 和解决 SLA 各自的达成率、平均时长、P90 时长、违约数 | 用 P90（而非仅均值）暴露长尾问题，避免"均值达标但 10% 工单严重超标" |
| **按优先级拆分行** | P1-P4 各自的响应/解决达成率、违约数、平均解决时长 | 高优工单的 SLA 往往是客户最关注的，需要独立看 |
| **SLA 达成率趋势** | 三条线：响应达成率 + 解决达成率 + P1/P2 达成率 | 判断 SLA 是在改善还是恶化，三条线差异揭示结构性问题 |
| **优先级对比图** | 分组柱状图：每个优先级的响应 vs 解决达成率 | 一眼看出哪个优先级、哪个环节是短板 |
| **违约分布饼图 x2** | 按优先级分布 + 按类别分布 | 快速定位违约集中在哪个优先级和哪个技术领域 |
| **类别/人员风险排名** | 违约最多的 Top 5 类别 + Top 5 处理人 | 定位到具体领域和具体人员，支撑改进决策 |
| **违约样本表** | 最多 12 条违约工单明细（编号、优先级、时长、违约类型） | 从统计到个案——看到具体哪条工单违约最严重 |

#### Tab 3：事件分析

**定位**：事件管理的核心分析视图，覆盖工单量、MTTR、分布和趋势。

**设计思路**：事件管理是 ITSM 中频率最高、客户感知最强的流程。SDM 需要回答的核心问题是：故障多不多（单量）、修得快不快（MTTR）、高优占比高不高（P1/P2）、趋势如何。Tab 的结构围绕这四个问题展开，用趋势图展示变化方向，用分布图展示结构，用明细表落到个案。

| 内容类型 | 内容 | 业务价值 |
|---------|------|---------|
| **6 张 MetricCard** | 事件总数、P1/P2 数、未解决数、SLA 达成率、P1/P2 MTTR、平均 MTTR | 事件管理的 6 个核心 KPI 一目了然 |
| **单量趋势图** | 组合图：周度事件单量（柱）+ SLA 达成率（线） | 单量与质量的关联观察——单量激增时 SLA 是否同步恶化 |
| **MTTR 趋势图** | 双线图：整体 MTTR + P1/P2 MTTR | 区分整体修复效率和关键故障修复效率 |
| **优先级分布饼图** | P1/P2/P3/P4 占比 | 高优占比异常（>15%触发 warning）是预警信号 |
| **类别分布饼图** | Top 8 事件类别 | 定位故障集中在哪个技术领域，支撑资源调配 |
| **处理人工作量 Top 10** | 各处理人的事件数排名 | 发现负载不均衡——个别人员过载既是质量风险也是流失风险 |
| **事件样本表** | Top 15 事件明细（含 SLA 达标标记 ✓/✗） | 落到具体工单，支撑复盘和改进讨论 |

#### Tab 4：变更分析

**定位**：变更管理的质量与风险分析视图。

**设计思路**：变更管理的核心矛盾是"速度 vs 安全"。SDM 需要知道变更成不成功、紧急变更多不多、变更有没有引发故障、计划执行得好不好。这个 Tab 在成功率的基础上，增加了"变更致事件"和"计划偏差"两个维度的分析，帮助团队从"变更能做完"走向"变更做得好"。

| 内容类型 | 内容 | 业务价值 |
|---------|------|---------|
| **4 张 MetricCard** | 变更总数、成功率、紧急变更数、引发事件的变更数 | 变更质量的 4 个核心指标 |
| **成功率趋势图** | 组合图：变更数量（柱）+ 成功率（线）+ 引发事件数（柱） | 三维观察：量、质量、风险随时间的变化 |
| **变更等级分布** | 饼图：Normal/Emergency 等类型占比 | 紧急变更占比过高（>15%）是流程成熟度不足的信号 |
| **类别分布堆叠柱图** | 按类别展示成功/失败的堆叠比例 | 定位哪类变更失败率最高，聚焦改进 |
| **引发故障分布** | 引发事件的变更按风险等级分布 | 量化变更风险：不是所有变更都引发故障，高风险变更是重点管控对象 |
| **计划偏差堆叠柱图** | 按变更类型展示提前/按时/延期比例 | 变更计划能力评估——延期率高意味着排期不准确或执行不可控 |
| **失败/回退样本表** | 最多 10 条失败变更明细 | 失败变更是复盘的起点，需要落到个案 |

#### Tab 5：请求分析

**定位**：服务请求的履约质量与客户满意度分析视图。

**设计思路**：服务请求是客户日常接触最多的 ITSM 触点（密码重置、权限申请、设备申请等），高频且直接感知。分析重点不是"有没有做完"（大部分请求都会完成），而是"做得快不快"（SLA）和"客户满不满意"（CSAT）。这个 Tab 将客观指标（SLA、耗时）与主观指标（满意度）并重，并通过部门维度发现满意度洼地。

| 内容类型 | 内容 | 业务价值 |
|---------|------|---------|
| **4 张 MetricCard** | 请求总数、已完成数、SLA 达成率、平均满意度 | 请求管理的 4 个核心 KPI |
| **单量趋势图** | 组合图：请求单量（柱）+ 平均满意度（线） | 单量与满意度的关联——请求激增时满意度是否下降 |
| **SLA 与耗时对比图** | 组合图：各类别的平均耗时（柱）+ SLA 达成率（线） | 识别"耗时长但 SLA 达标"和"耗时短但 SLA 不达标"的矛盾类别 |
| **请求类型分布** | 饼图：Top 6 请求类型 | 了解请求结构，支撑自动化优先级决策（高频请求考虑自动化） |
| **部门请求排名** | 柱状图：Top 8 部门请求数 | 发现哪些部门是服务消费大户，支撑资源规划 |
| **满意度分布饼图** | 按满意度分段分布（非常满意/满意/不满意/非常不满意） | 满意度不只是均值——均值 3.8 可能掩盖 30% 的不满意用户 |
| **高频请求目录** | 柱状图：Top 8 请求类别 | 高频类别是流程优化和自动化的首选目标 |
| **低满意度样本表** | 满意度 ≤ 3 的请求明细 | 不满意客户的工单是改进的直接线索 |

#### Tab 6：问题分析

**定位**：问题管理的成熟度分析视图——从被动救火到主动根因治理。

**设计思路**：问题管理在 ITSM 中是"重要但不紧急"的流程——它不直接处理客户故障，但决定着故障是否反复发生。SDM 需要知道问题关闭得快不快、根因分析（RCA）做得彻不彻底、有没有永久修复方案。这个 Tab 的设计重点不是"问题有多少"，而是"问题有没有被根治"，通过"解决方案健康度"堆叠图直接展示永久修复 vs 临时方案 vs 未解决的比例。

| 内容类型 | 内容 | 业务价值 |
|---------|------|---------|
| **4 张 MetricCard** | 问题总数、已关闭数、已完成 RCA 数、已知错误数 | 问题管理流程的关键产出指标 |
| **问题单量趋势图** | 组合图：问题单量（柱）+ 累积未解决数（线） | 累积未解决线持续上升 = 团队在问题管理上入不敷出 |
| **根因类型分布** | 饼图：Top 6 根因类别 | 根因集中在哪——技术缺陷？流程缺陷？人为错误？决定改进方向 |
| **引发故障数排名** | 柱状图：各类根因引发的事件数 | 量化每种根因的实际影响——根因频率不高但引发大量事件的类型最危险 |
| **状态分布** | 饼图：问题状态分布 | Open/In Progress/Closed 的比例反映问题管理的吞吐效率 |
| **解决方案健康度** | 堆叠柱图：永久修复 / 有临时方案 / 未解决 | 核心指标——问题管理的目标是根治而非关闭。"有临时方案但无根治"是技术债的来源 |
| **模块薄弱点分析** | 分组柱图：根因类别 × 配置项交叉分析 | 定位系统中最薄弱的模块——哪个系统、什么类型的缺陷最多 |
| **未关闭问题表** | 最多 10 条未关闭问题明细 | 需要关注的老问题——尤其是高关联事件数的问题 |

#### Tab 7：跨流程关联

**定位**：打破四大流程的信息孤岛，揭示流程间的因果链路和风险传导路径。

**设计思路**：ITSM 四大流程不是孤立的——一次变更可能导致故障，一个故障可能暴露问题，一个问题可能关联多条请求。传统的单流程仪表盘无法呈现这些关联。这个 Tab 的核心价值是回答"变更致事件率是多少"、"变更和事件在时间上是否同步"、"哪个系统最脆弱"、"请求激增是否预示故障"。通过 7×24 热力图、气泡图、时间重叠图等可视化手段，将跨流程的因果和时序关系显性化。

| 内容类型 | 内容 | 业务价值 |
|---------|------|---------|
| **4 张 MetricCard** | 变更致事件率、48h 窗口 P1/P2 事件数、请求-事件比、系统脆弱性评分 | 四个跨流程的核心风险指标 |
| **变更致事件趋势** | 组合图：变更数量 + 致事件 P1/P2 数 | 变更管控是否有效的时序验证 |
| **变更风险热力图** | 7×24 热力图：变更密度 + 事件热点的时段分布 | 发现高危时段——变更和事件在什么时间点重叠 |
| **系统脆弱性气泡图** | 气泡图：横轴事件总数、纵轴老化天数、气泡大小为未关闭问题数 | 一眼识别最脆弱的系统——事件多、老化久、问题积压 |
| **请求与事件时间重叠** | 组合图：周度请求数 vs 事件数 | 请求激增是否领先于事件激增——如果是，请求量是故障的先行指标 |
| **变更致事件明细表** | 具体变更 → 关联 P1/P2 事件的链路 | 从统计到个案：哪个变更导致了什么故障 |
| **老化问题清单** | 按老化天数排序的未关闭问题 | 长期未解决的问题 = 持续的系统性风险 |
| **请求激增预警表** | 各类别的请求周环比增长 + 同期事件数 | 主动预警：某类请求突然激增时，可能预示服务异常 |

#### Tab 8：团队效能

**定位**：将四类 ITSM 流程的指标映射到人员维度，支撑绩效管理和负载均衡决策。

**设计思路**：运维服务的质量最终由人决定。这个 Tab 回答三个问题：**产量**（每个人做了多少）、**效率**（做得快不快）、**质量**（做得好不好）。通过负载分布图展示均衡性，通过热力图暴露技能缺口（某人在某领域耗时远高于团队平均 = 技能不足），通过气泡图和雷达图综合评估个人素质。三个排名表（救火先锋、技术债务解决、高风险变更执行）将不同角色的贡献显性化。

| 内容类型 | 内容 | 业务价值 |
|---------|------|---------|
| **8 张 MetricCard** | 人均处理量、积压工单数、平均交付耗时、SLA 达标率、变更实施速度、一次性成功率、用户满意度、问题根治率 | 团队效能的 8 维概览——产量、效率、质量全覆盖 |
| **团队负载分布** | 分组柱图：Top 10 人员的各域工单分布 | 发现负载不均衡——某人事件多但变更少可能意味着技能单一或分工不合理 |
| **工单流转热力图** | 热力图：人员 × 类别的平均耗时矩阵 | 暴露技能缺口——某人在某类别耗时远高于团队平均 = 培训需求 |
| **SLA vs 满意度气泡图** | 气泡图：横轴 SLA 达标率、纵轴满意度、气泡大小为工单量 | 发现"高效但低满意度"或"低效但高满意度"的异常人员 |
| **个人综合素质雷达图** | Top 5 人员的 5 维雷达（速度、产量、质量、满意度、难度） | 综合评估个人能力，避免单一指标误导 |
| **救火先锋 Top 10** | P1/P2 事件处理数最多的人员 | 识别关键人才——P1/P2 处理集中度高 = 单点风险（bus factor） |
| **技术债务解决 Top 10** | 根治问题数最多的人员 | 将"根治问题"这类低调但高价值的工作显性化 |
| **高风险变更执行 Top 10** | 紧急/高风险变更执行最多的人员 | 识别变更执行的关键人员和其成功率 |

## 4. 模块二：AI Native Report Agent

### 4.1 模块概述

Report Agent 是一个 **理解自然语言分析意图、自主选择分析工具、生成数据驱动分析报告** 的 AI Agent。它是 ITSM 运营数据与业务决策之间的智能桥梁。

**代码位置**：`gateway/agents/report-agent/`

**技术栈**：Goose Agent Runtime + qwen3.5-27B + MCP (Python stdio)

### 4.2 为什么用 Agent 而不是模板化报告

| 维度 | 模板化报告 | AI Agent（我们的方案） |
|------|----------|---------------------|
| 灵活性 | 固定格式、固定维度 | 按需分析任意维度组合 |
| 交互性 | 生成后不可追问 | 多轮对话、逐步下钻 |
| 洞察深度 | 数据罗列 | 基于风险雷达的主动洞察 |
| 个性化 | 所有人看同一份 | 针对具体问题生成针对性建议 |
| 门槛 | 需理解报告结构和数据源 | 零门槛，自然语言提问 |

### 4.4 核心能力

#### 能力 1：意图驱动的工具选择

Agent 的工具不是按数据域命名（如"获取 SLA 数据"），而是按 **分析意图** 命名（如"分析 SLA 达成率"）。工具名即分析概念，模型看到工具名就知道适用场景：

| 用户说什么 | Agent 选择什么工具 |
|-----------|-----------------|
| "SLA 达成率怎么样？" | `analyze_sla_rate` |
| "最近故障工单量趋势？" | `analyze_incident_volume(by_time=true)` |
| "平均修复时间按人员看？" | `analyze_mttr(by_resolver=true)` |
| "变更成功率怎么样？" | `analyze_change_success_rate` |
| "哪个部门满意度最低？" | `analyze_request_performance(by_department=true)` |
| "问题根因分布？" | `analyze_problem_metrics(by_root_cause=true)` |
| "团队绩效排名？" | `analyze_workforce_performance` |
| "给我月度概览" | `get_all_metrics` |

**完整工具清单（11 个）**：

| 工具名 | 分析意图 | 调用的 BI 端点 |
|--------|---------|--------------|
| `get_all_metrics` | 全域概览 | GET /metrics/{domain} x8 |
| `analyze_sla_rate` | SLA 达成分析 | GET /metrics/sla + trends |
| `analyze_incident_volume` | 事件工单量分析 | GET /metrics/incidents + query |
| `analyze_mttr` | 平均修复时间分析 | GET /metrics/incidents + query |
| `analyze_change_success_rate` | 变更成功率分析 | GET /metrics/changes + query |
| `analyze_request_performance` | 服务请求绩效分析 | GET /metrics/requests + query |
| `analyze_problem_metrics` | 问题管理分析 | GET /metrics/problems + query |
| `analyze_workforce_performance` | 团队绩效分析 | GET /metrics/workforce |
| `query_tickets` | 工单明细查询 | POST /data/{domain}/query |
| `compute_metric` | 自定义指标计算 | POST /data/{domain}/query |
| `trace_ticket_lineage` | 跨流程关联追溯 | GET /data/{domain}/lineage |

**达成效果**：用户不需要知道数据在哪个系统、用什么 API，只需用业务语言提问。

#### 能力 2：智能路由与参数默认值

Agent 调用工具时不需要指定所有参数，系统为每个分析场景预设了合理默认值：

- 调 `analyze_sla_rate` → 默认返回整体达成率 + 按优先级拆分
- 调 `analyze_workforce_performance` → 默认返回 Top 10 人员
- 只有用户明确要求额外维度时，Agent 才需要设置对应参数

**设计考量**：工具设计对 27B 参数模型友好。通过"默认够用、按需扩展"的模式，降低模型的参数决策负担，保证基本交互质量。

#### 能力 3：数据驱动的风险洞察（风险雷达）

风险雷达是一个**多层协作**的风险识别体系，涉及代码实现和 LLM 推理两个层面：

**第一层：BI 引擎自动计算（代码实现）**

BI Java 后端的 `BusinessIntelligenceMetricsService.buildExecutiveRiskItems()` 在 Executive 域指标计算时自动执行风险检测，通过 API 返回结构化的风险项（含 criticalCount / warningCount / attentionCount / topRisks 字段）。当前实现的规则：

| 规则 ID | 风险信号 | 判定条件 | 严重度 |
|---------|---------|---------|--------|
| `change-failure` | 变更失败 | 变更失败数 >= 5 | **CRITICAL** |
| `problem-closure` | 问题关闭率低 | 关闭率 < 55% | WARNING |
| `request-open` | 请求积压 | 未完成请求 >= 15 | WARNING |
| `change-incident` | 变更致事件 | 变更致事件率 >= 10% | WARNING |
| `request-csat` | 满意度偏低 | CSAT < 3.8 | ATTENTION |
| `incident-sla` | SLA 违约 | 存在 SLA 违约 | ATTENTION |
| `problem-open` | 问题积压 | 未关闭问题 >= 20 | ATTENTION |

> **注意**：这部分规则与 AGENTS.md 中描述的 R001-R010 存在差异，属于待对齐项。

**第二层：LLM 风险判断（Prompt 引导）**

AGENTS.md 中定义了 R001-R010 共 10 条风险雷达规则作为 LLM 的判断参考。Agent 在获取到指标数据后，结合 Prompt 中的规则描述，自主识别风险信号并给出建议。R001-R010 规则：

| 规则 | 风险信号 | 判定条件 | 严重度 |
|------|---------|---------|--------|
| R001 | SLA 严重违约 | Incident SLA Rate < 70% | **CRITICAL** |
| R002 | 关键故障恢复慢 | P1/P2 MTTR > 24h | **CRITICAL** |
| R003 | 变更质量差 | Change Success Rate < 80% | WARNING |
| R004 | 变更引发故障多 | 变更致事件率 > 10% | WARNING |
| R005 | 用户满意度低 | Request CSAT < 3.0 | WARNING |
| R006 | 问题积压严重 | Problem 关闭率 < 50% | ATTENTION |
| R007 | 高优工单占比异常 | P1/P2 占比 > 20% | ATTENTION |
| R008 | 紧急变更过多 | 紧急变更占比 > 15% | ATTENTION |
| R009 | 已知错误无规避方案 | 问题已知错误无 workaround | ATTENTION |
| R010 | 人员过载 | 人员积压 > 20 单 | ATTENTION |

**第三层：Python 分析器（离线报告）**

`business-intelligence/scripts/analyzer.py` 中硬编码了 R001-R009 的阈值检查逻辑（R010 未实现），用于离线 Excel 报告生成，不参与 Agent 在线流程。

**达成效果**：Agent 不只是复述数据，而是结合引擎预计算的风险项和 LLM 自身的判断能力，主动识别风险、提出建议，从"报表工具"进化为"分析助手"。

**待改进**：三层风险规则目前存在不一致（Java 后端 vs AGENTS.md vs analyzer.py 的规则 ID、阈值均不同），后续需要对齐统一。

#### 能力 4：多语言报告生成

Agent 自动匹配用户语言：

- 用户用中文提问 → 生成中文报告（标题、分析、建议均为中文）
- 用户用英文提问 → 生成英文报告
- 中途切换语言 → 后续响应跟随新语言

报告格式遵循统一模板：

```markdown
# {标题}
**Period**: {起止时间} | **Sources**: {数据来源}

## Key Metrics
| Metric | Value | Status |

## Top Findings
1. {数据驱动的洞察发现}

## Recommendations
1. {可操作建议}
```

报告内容的质量约束：
- 所有数字和结论 100% 来自工具返回数据，**零编造**
- 包含分析期间和数据来源说明
- 聚焦关键发现（2-3 条洞察/节），不超过 3000 字
- 给出 3-5 条可操作建议
- 报告以 Markdown 格式保存为文件，同时在对话中输出摘要

#### 能力 5：多轮对话下钻与工单追溯

Agent 支持多轮对话，逐步深入分析：

```
第 1 轮: "SLA 达成率怎么样？"
         → analyze_sla_rate → 发现 P1/P2 解决达成率仅 45%

第 2 轮: "按人员看呢，谁负责的违约最多？"
         → analyze_sla_rate(by_resolver=true) → 定位到张三

第 3 轮: "给我张三负责的违约工单明细"
         → query_tickets(filters=[assigned_to=张三, SLA=No], sort=resolution_time desc)
         → 返回具体工单列表，包含工单号、优先级、解决时长

第 4 轮: "这个 CHG-2026-0042 变更导致了哪些故障？"
         → trace_ticket_lineage(ticket_id=CHG-2026-0042)
         → 返回关联的事件工单链路
```

**达成效果**：从指标到工单的闭环——发现异常 → 定位具体工单 → 追溯根因。

### 4.5 MCP 桥接层设计

`bi-data-service` 是一个 Python MCP Server（stdio 传输），负责将 Agent 的工具调用翻译为 BI 引擎的 HTTP 请求。关键设计：

| 设计点 | 说明 |
|--------|------|
| **KPI 精简** | `get_all_metrics` 从每个域的完整响应中只提取顶层 KPI 字段，减少 token 消耗 |
| **趋势合并** | 多条趋势线（如响应 SLA + 解决 SLA + P1/P2 SLA）合并为单时间索引表 |
| **字段上下文注入** | 查询结果附带 `_context` 和 `_fields` 元数据，帮助 LLM 理解列含义 |
| **重复调用检测** | 追踪最近 10 次工具调用，连续 2 次相同调用后自动阻断，防止无限循环 |
| **结果截断** | 超过 20,000 字符的结果自动截断并提示缩小查询范围 |

## 5. 两个模块的协同价值

### 5.1 对运维服务的价值总结

| 价值维度 | 传统方式 | BI + Report Agent 方案 |
|---------|---------|----------------------|
| **运营感知时效** | 月报延迟 3-5 天，数据已过时 | 分钟级获取任意时间范围的分析报告 |
| **分析深度** | 依赖个人 Excel 能力，维度有限 | 8 域 × 7 维度 × 趋势 × 关联，任意组合 |
| **风险识别** | 靠人工经验发现异常 | 10 条风险雷达规则自动扫描，不遗漏 |
| **报告质量** | 格式不一、数据可能出错 | 标准化输出、数据 100% 可溯源、零编造 |
| **使用门槛** | 需理解数据结构和工具 | 自然语言提问，零学习成本 |
| **跨流程洞察** | 各流程孤立分析 | 变更→事件→问题→请求全链路追溯 |
| **决策支持** | 数据罗列，需要自己解读 | 结构化洞察 + 可操作建议 |

### 5.2 典型使用场景

**场景 1：月度运营汇报**

SDM 在月度汇报前，直接向 Report Agent 提问"给我上个月的运维运营概览"，Agent 自动获取全部 8 域指标、应用风险雷达、生成包含健康评分、KPI 总览表、Top 风险和建议的专业报告。从过去需要 2-3 天人工整理，缩短到 2 分钟内完成。

**场景 2：异常下钻排查**

SDM 发现 SLA 达成率偏低，通过多轮对话逐步下钻——先看按优先级拆分，发现是 P1/P2 拖后腿；再看按人员拆分，定位到具体负责人；最后查看该负责人的违约工单明细，找到根因。整个过程无需打开任何 Excel。

**场景 3：变更风险评估**

重大变更上线前，SDM 提问"最近变更导致的故障有哪些？"，Agent 自动分析变更致事件率趋势、追溯具体变更工单的关联事件链，给出变更风险建议。将事后救火变为事前预判。

### 5.3 协同工作流

```
用户提问
  │
  ├─ 简单查询 "这个月有多少事件工单？"
  │    └→ Agent → query_tickets(count) → 直接回答
  │
  ├─ 单域分析 "SLA 达成率怎么样？"
  │    └→ Agent → analyze_sla_rate → 指标 + 按优先级拆分 + 趋势 → 生成分析
  │
  ├─ 全域概览 "给我月度运营概览"
  │    └→ Agent → get_all_metrics → 8 域 KPI + 风险雷达 → 生成报告
  │
  ├─ 追问下钻 "按人员看呢？"
  │    └→ Agent → analyze_*(by_resolver=true) → 定位到人 → 继续分析
  │
  └─ 工单追溯 "这个变更导致了什么故障？"
       └→ Agent → trace_ticket_lineage → 关联链路 → 根因分析
```

## 6. 演进路线

### 演进方向总览

```
Phase 1 (已完成)          Phase 2                   Phase 3
可计算底座                分析深化 + 技能化          SDM Copilot
──────────────     →     ─────────────────     →     ────────────
8 域指标底座              同比/环比 + 归因分析       主动巡检 + 风险预警
11 个 MCP 工具            工具优化 + 多轮对话增强     周期报告自动送达
自然语言报告生成           最佳实践 Skills            分析模式学习
Excel 文件数据源          数据源适配器（保持解耦）     SDM 偏好记忆
```

**核心原则：保持 BI 模块与上游 ITSM 系统的解耦。** BI 模块通过标准化 schema 文件接收数据，不直接依赖任何 ITSM 系统的 API。数据获取由外部适配器完成，BI 模块专注于分析和计算。

### Phase 1（当前已完成）：可计算底座 + Agent 基础交互

**BI 分析引擎**：

- 8 域聚合指标 + 多维度下钻 + 趋势 + 查询 + 关联
- 仪表盘前端 8 Tab 可视化
- 标准化 schema v1 数据契约，适配多平台导出

**Report Agent**：

- 11 个意图驱动工具 + 智能路由
- 中文/英文报告生成
- 风险雷达（代码实现 + Prompt 引导）
- 工单明细查询与跨流程追溯

### Phase 2：分析深化 + MCP 工具优化 + Skills 技能化

在 Phase 1 的"能查到数据、能生成报告"基础上，Phase 2 要做到"分析有深度、工具有精度、报告有套路"。

**分析能力深化**：

| 能力 | 说明 | 业务价值 |
|------|------|---------|
| 同比/环比自动计算 | 在趋势数据中内置周期对比，返回变化幅度和方向 | "SLA 比上月改善了还是恶化了？" 不需要 SDM 自己对比两份报告 |
| 异常归因分析 | 当指标恶化时，自动定位贡献最大的维度（如"SLA 下降主要因为 P2 事件的解决 SLA 从 95% 跌到 70%"） | 从"发现问题"到"解释问题"，减少 SDM 的下钻轮次 |
| 跨流程根因链路 | 强化变更→事件→问题的因果链路分析，输出结构化的根因路径 | 支撑复盘：不只能看到变更导致了故障，还能看到背后的根因类别和修复建议 |
| 多租户数据隔离 | 按客户/项目维度独立分析，BI 引擎层支持数据过滤 | 不同客户/项目独立运营分析，互不干扰 |

**MCP 工具优化**：

| 优化方向 | 说明 |
|---------|------|
| 减少工具调用轮次 | 合并高频组合查询（如 SLA + MTTR 一次调用返回），降低多轮交互中的 token 消耗和延迟 |
| 结果结构优化 | 按模型推理特点优化返回格式，突出关键数字和变化信号，压缩无关细节 |
| 工具描述精炼 | 基于实际调用日志持续优化工具描述，提高模型工具选择准确率 |
| 重复调用防护增强 | 当前已有基础的去重机制，后续增加"相似意图检测"，避免模型用不同参数反复问同一个问题 |

**Report Agent Skills——最佳实践报表方案**：

为 Report Agent 引入 Skills 机制，每个 Skill 封装一套"分析框架 + 输出规范"——告诉 Agent 应该关注哪些指标、按什么维度下钻、风险阈值怎么设，以及最终产出什么格式的报告。

Skill 支持两种输出模式：

- **交互式分析**：SDM 用自然语言触发 Skill，Agent 按框架逐步分析，支持追问下钻，输出 Markdown 报告
- **固定格式报表导出**：按预设模板生成结构化报表（如客户月度汇报用的 PPT/Excel/Word），满足特定场景对格式和内容的严格要求。模板定义章节结构、必含指标、图表类型和排版规范，Agent 负责填充数据和分析内容

```
Skill 触发方式
├── 对话触发:  "帮我做一份月度运营汇报"
│     └→ Agent 按月度概览 Skill 的分析框架执行 → 输出 Markdown
│
└── 报表导出:  "导出客户月度汇报报表"
      └→ Agent 按 Skill 定义的模板结构 → 填充数据 + 图表 → 导出固定格式文件
```

| Skill | 分析框架 | 输出模式 | 业务场景 |
|-------|---------|---------|---------|
| **月度运营概览** | 全域 KPI 扫描 → 风险雷达 → Top 3 关注项 → 改进建议 | 交互式 + 月度汇报模板导出 | SDM 月度运营汇报、客户月度服务质量报告 |
| **SLA 达成率专项** | 整体达成率 → 按优先级/类别/人员拆分 → 违约归因 → 趋势判断 | 交互式 + SLA 专项报告模板导出 | SLA 违约排查、合同续签 SLA 附件 |
| **变更风险复盘** | 变更致事件率 → 48h 窗口关联 → 失败变更明细 → 高危变更识别 | 交互式 | 重大变更后的风险评估 |
| **团队绩效评估** | 产量/效率/质量三维 → 负载均衡度 → 技能缺口热力图 → 排名 | 交互式 + 绩效报告模板导出 | 团队绩效回顾与资源调配 |
| **跨流程关联分析** | 变更→事件→问题链路 → 系统脆弱性评分 → 请求激增预警 | 交互式 | 季度系统性风险排查 |
| **问题治理成熟度** | 关闭率/RCA 率 → 解决方案健康度（永久修复 vs 临时方案） → 技术债 | 交互式 + 问题治理报告模板导出 | 问题管理流程改进、季度治理评审 |

**数据源适配器（保持解耦）**：

BI 模块继续通过标准化文件接收数据，不直接接入 ITSM 系统 API。数据获取由独立的适配器层完成：

```
ITSM 系统 A ──→ 适配器 A ──→  standardized Excel ──→  BI Engine
ITSM 系统 B ──→ 适配器 B ──→  standardized Excel ──→  BI Engine
手动导出     ──────────────→  standardized Excel ──→  BI Engine
```

适配器只需实现 `itil-ticket-export-schema-v1` 规范的 Excel 输出，BI 模块无需感知上游系统差异。后续可开发定时同步脚本自动拉取数据并刷新 BI 缓存，但 BI 模块本身保持无状态、无外部依赖。

### Phase 3：SDM Copilot——从被动分析到主动智能

Phase 3 的目标是让 Report Agent 从"SDM 问什么分析什么"的工具，进化为"主动帮 SDM 发现问题、推送洞察、管理运营节奏"的 **SDM Copilot**。

**主动巡检与风险预警**：

| 能力 | 说明 |
|------|------|
| 定期自动巡检 | Agent 按预设周期（每日/每周）自动执行全量指标扫描，无需 SDM 手动触发 |
| 阈值告警推送 | 当关键指标（SLA 达成率、MTTR、变更致事件率等）突破预设阈值时，主动推送风险提醒 |
| 趋势恶化预警 | 不仅看绝对值，还看趋势——连续 3 周恶化的指标即使未触线也应预警 |
| 异常模式检测 | 识别非常规模式（如某类事件突然激增、某人员积压异常增长） |

**周期报告自动送达**：

| 能力 | 说明 |
|------|------|
| 自动生成周报/月报 | Agent 按计划自动生成周期运营报告，SDM 无需手动触发 |
| 报告批注与交互 | SDM 在报告上标注关注点，Agent 后续分析自动聚焦这些维度 |
| 历史报告对比 | "这个月的报告和上月相比有什么变化？" 自动高亮差异项 |

**SDM 偏好记忆与学习**：

| 能力 | 说明 |
|------|------|
| 关注偏好学习 | Agent 记住 SDM 最常问的指标和维度，主动推送相关洞察而非无关信息 |
| 分析模式沉淀 | 将 SDM 的典型分析路径（如"先看 SLA → 再按人员下钻 → 看违约样本"）固化为可复用的分析流 |
| 上下文延续 | 跨会话记住上次讨论的结论，下次对话时自动关联："上次会议讨论的 SLA 改善措施效果如何？" |

## 7. 关键技术决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 工具命名方式 | 按分析意图命名（analyze_sla_rate）而非数据域命名（get_sla_data） | 降低 27B 模型的工具选择负担，意图映射更直观 |
| MCP 传输方式 | stdio（非 SSE/HTTP） | 与 Goose Agent Runtime 集成简单，进程隔离天然安全 |
| LLM 模型选择 | qwen3.5-27B | 在工具选择准确率和推理能力之间取得平衡，成本可控 |
| 温度参数 | 0.2（低） | 分析报告需要确定性输出，减少幻觉和随机性 |
| 数据加载方式 | 标准化 Excel 文件 + 缓存 + 外部适配器 | 保持 BI 模块与 ITSM 系统解耦，适配器层负责数据获取，BI 专注分析 |
| SLA 计算位置 | BI 引擎侧计算，不依赖原始标记 | 统一计算逻辑，确保仪表盘和 Agent 结果一致 |
| 指标权重 | 硬编码配置 | Phase 1 简化实现，Phase 2 支持按客户/项目自定义 |
